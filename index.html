Part 1 · System Architecture

02Firmware

RTOS: Zephyr + nRF Connect SDK - BLE stack, DT variants, host CI (native_sim), MCUboot DFU.

Portable C11 core

No Zephyr types, no malloc. Ring buffers, sequence numbers, drop counts. Full → overwrite oldest.

Thin port

~40 lines: lock + monotonic time. Host-testable under sanitizers.

BLE only pulls

Drivers never call BLE. Radio thread drains batches on connection events.

PPR? Measure first

Optional: sample on co-processor, M33 sleeps. Cost: IPC + dual image. DMA-only may be enough - decide on EVK + PPK.

03Hardware

Rigid-flex islands (MCU/radio, PPG, battery). Height matters. One shared I²C is fragile - hung slave kills all sensing. Rev plan: PPG on its own SPI; I²C recovery + per-sensor power gate.

LabWired twin: firmware reads real sensor ID registers over I²C (not assumed “present”).

Antenna

Finger detunes RF. Keep-out; tune on finger phantom, not free space only.

PPG rail

Own filtered analog supply. BLE TX noise must not look like pulse.

Rail gating

Switch digital / analog / motor rails. Standby µA add up on a week budget.

Temp honesty

Reads skin + self-heat. Use for trends, not core body temp.

04BLE

SIG Device Info + Battery + Time. Custom Ring Data: capability, stream control, data notify, config, control point, backfill.

Batch, don't spam. 50 Hz is easy bandwidth; wakeups kill the cell. Fill negotiated MTU (~16 IMU samples / notify). Derive batch size from MTU (iOS ≠ Android).

ModeConn intervalIntent
Week400–1000 msBackground log, low radio duty
Research15–50 msLive stream, ~7× radio energy

Peripheral requests params. LESC + rotating private address. Offline log + backfill. Signed dual-slot DFU.

05Power

TargetBudget @ 20 mAh
1 week~119 µA average
24 hours~833 µA average

Cell size first

Shipping rings often use ~10 mAh. Then week budget ~61 µA and a ~108 µA design lasts < 4 days. Measure the cell before promising a week.

PPG LEDs dominate. Cut duty first: sample when still, min LED for SNR, shorter pulses. Then batch BLE. MCU micro-opts last.

Rough research mode ~878 µA → ~22.8 h - misses 24 h by ~5% on estimate. Trim PPG / gyro before claiming.

06Expansion & modes

Sensors in a static table → BLE capability at build time. New sensor = table row + driver; old apps ignore unknown types.

Part 3 · Debugging

Method: cheapest discriminating test first · reproduce before changing · bisect (electrical / FW / algo) · one variable at a time.

08Issue A - PPG bad when BLE is busy

Scales with radio → power rail, RF/EMI, or timing (CPU/bus). Partition with two tests: bench-power the PPG; CPU busy-loop, radio off.

Twin: timing path only

Models CPU starvation → PPG FIFO overflow under “BLE busy.” Does not model analog rail coupling.

09Issue B - battery ~40% short

First: was “expected” a measured budget or a spreadsheet? Audit the model. Often that is the bug.

10Issue C - 12% FPY fail

12% is meaningless until Pareto. Then: bad product or bad test? Retest fails.

Highest leverage: log measured values per step per serial, not pass/fail only. DFT, traceability, shielded RF, stage board test before battery/enclosure.

11Assumptions & risks

Assumptions

20 mAh usable (likely high). Currents are estimates. PPG is #1 load. On-chip log memory enough.

Risks

Antenna on finger; research < 24 h; false temp; motion-ruined PPG; dual-core version skew.

  • PPR vs DMA energy - PPK both paths
  • Real PPG mAh at usable SNR on real fingers
  • Gating vs standby per sensor; assembled leakage
  • Worn antenna TRP/TIS
